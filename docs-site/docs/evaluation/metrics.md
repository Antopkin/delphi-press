# Evaluation Metrics

## Brier Score: основная метрика

Калибровка вероятностных предсказаний измеряется стандартной метрикой **Brier Score** (BS), изначально разработанной для оценки метеорологических прогнозов.

$$BS = \frac{1}{N} \sum_{i=1}^{N} (f_i - o_i)^2$$

где $f_i \in [0,1]$ — предсказанная вероятность события $i$, $o_i \in \{0,1\}$ — фактический исход (1, если событие произошло, 0 — в противном случае).

В контексте Delphi Press предсказываемым событием является наличие освещения прогнозируемой темы в целевом медиаиздании на заданный горизонт времени (1, 3 или 7 дней). Вероятность $f_i$ берётся из поля `FinalPrediction.confidence`, а бинаризация исхода $o_i$ определяется как:

$$o_i = \begin{cases}
1, & \text{если } \texttt{TopicMatch} > 0 \\
0, & \text{иначе}
\end{cases}$$

Источники для ground truth — актуальные RSS-потоки 16 источников проекта (ТАСС RU/EN, РИА Новости, Интерфакс, BBC, Al Jazeera, Guardian и др.) и Wayback Machine CDX API для восстановления снимков веб-страниц за прошлые даты.

Для оценки навыка система использует **Brier Skill Score**:

$$BSS = 1 - \frac{BS}{0.25}$$

где 0.25 — базовая ошибка случайного угадывания (всегда вероятность 50%). BSS интерпретируется как относительное улучшение: $BSS = 0$ означает производительность на уровне случайности, $BSS = 1$ — идеальное предсказание.

### Benchmark Ориентиры

| Система/уровень | Brier Score | BSS | Контекст |
|---|---|---|---|
| Случайное угадывание | 0.250 | 0.00 | Базовая линия (всегда 50%) |
| Polymarket (prediction market) | ≈0.187 | ≈0.25 | Реальные деньги, многоуровневые участники[^1] |
| Metaculus-сообщество (AI-вопросы) | 0.182 | ≈0.27 | Краудсорсированные прогнозы (self-reported) |
| Good Judgment Project (год 2) | ≈0.140 | ≈0.44 | Обученные суперпрогнозисты[^2] |
| **Delphi Press v1.0 (цель)** | **<0.20** | **>0.20** | Уровень prediction market |
| GPT-4.5 (ForecastBench, окт 2025) | 0.101 | ≈0.60 | Лучший LLM на сегодня [ForecastBench] |
| **Delphi Press v2.0 (цель)** | **<0.12** | **>0.52** | Уровень обученных прогнозистов |
| Суперпрогнозисты (ForecastBench) | 0.068–0.086 | 0.66–0.73 | Долгосрочный ориентир[^2]; числа из ForecastBench |

[^1]: Clinton & Huang (2024): Polymarket accuracy study, Vanderbilt University
[^2]: Tetlock & Gardner (2015): Superforecasting: The Art and Science of Prediction, Crown Publishers

!!! note "Статус внедрения"
    Пилотная оценка требует сбора ground truth через автоматизацию запроса к Wayback Machine CDX API и парсинга RSS-снимков. Пилотный тест спланирован на объёме 50 prediction runs по 3 горизонтам, что даст примерно 150–350 пар событие-исход. Статистическая мощность при N=50 обеспечивает 95%-й доверительный интервал ±0.03–0.05 вокруг BS. **Стоимость пилота: менее \$1** (только LLM-вызовы для граничных случаев TopicMatch; BERTScore и Wayback API бесплатны).

## Композитная оценка качества

Так как Brier Score оценивает лишь калибровку вероятностей, а не качество самого предсказанного текста, для полной оценки качества введена взвешенная композитная метрика:

$$\text{CompositeScore} = 0.40 \times \text{TopicMatch} + 0.35 \times \text{SemanticSim} + 0.25 \times \text{StyleMatch}$$

### Компоненты CompositeScore

| Компонент | Диапазон | Вычисление | Вес |
|---|---|---|---|
| TopicMatch | {0.0, 0.5, 1.0} | Тема совпадает с реальностью | 0.40 |
| SemanticSim | [0.0, 1.0] | BERTScore F1 vs реальный заголовок | 0.35 |
| StyleMatch | [0.0, 1.0] | LLM-as-judge оценка стилевого соответствия | 0.25 |

### TopicMatch: Трёхступенчатый алгоритм

**Шаг 1: Keyword screening** (< 5 мс)

- Извлечение ключевых слов из предсказанного заголовка
- Фильтрация кандидатов из ground truth

**Шаг 2: BERTScore** (~50 мс, локально)

- Вычисление контекстного сходства через многоязычную модель `xlm-roberta-base` (для кроссъязычных пар)

**Шаг 3: LLM-арбитр** (только для пограничных случаев $0.55 \leq F1 < 0.60$)

- ~10% случаев
- Экономит ~90% LLM-вызовов

### Пороговые значения TopicMatch

- $F1 \geq 0.78$ ⟹ TopicMatch = 1.0 (полное совпадение)
- $0.60 \leq F1 < 0.78$ ⟹ TopicMatch = 0.5 (верная тема, но неточный фрейминг)
- $0.55 \leq F1 < 0.60$ ⟹ LLM-арбитр (пограничные случаи)
- $F1 < 0.55$ ⟹ TopicMatch = 0.0 (промах)

!!! warning "Статус реализации"
    Трёхступенчатый алгоритм TopicMatch (keyword screening → BERTScore → LLM-arbiter) ещё не реализован в `src/eval/`. Пороги определены в спецификации, реализация запланирована.

### SemanticSim

F1-метрика BERTScore против лучшего совпадающего реального заголовка в день предсказания. Для монолингвальной оценки используются языкоспецифичные модели:

- `DeepPavlov/rubert-base-cased` для русского
- `roberta-large` для английского

### StyleMatch

Оценка по шкале 1–5 (затем нормализуется на [0,1] делением на 5) через LLM-as-judge. Критически: для избежания self-enhancement bias используется **отличная модель** от генератора заголовков. В стеке Delphi Press это Claude Opus 4.6 в роли судьи.

### Интерпретация CompositeScore

- **≥ 0.70** — отличный прогноз
- **0.50–0.69** — хороший
- **0.30–0.49** — частичное попадание
- **< 0.30** — промах

!!! warning "Известные ограничения"
    Пороги BERTScore (0.78, 0.60) откалиброваны на задачах перевода и суммаризации, но не валидированы специально для русскоязычных новостных заголовков длиной 5–15 слов. Рекомендуется ручная аннотация 20–30 примеров в начале пилота. Покрытие Wayback Machine неоднородно: ТАСС и РИА Новости имеют хорошее архивное покрытие, Коммерсант и Ведомости — частичное.

## Walk-Forward Валидация Informed Problem

Для доказательства того, что информированный консенсус бетторов Polymarket действительно даёт сигнал, применяется протокол walk-forward валидации с непересекающимися окнами. Это предотвращает look-ahead bias.

!!! note "Важное примечание о метрике"
    В отличие от теоретического BSS из раздела выше (baseline = random 0.25), **walk-forward BSS использует raw market price как baseline**: $BSS_{wf} = 1 - BS(\text{informed}) / BS(\text{raw market})$.

### Протокол

- **Burn-in:** 180 дней исторических данных для обучения профилей (безопасный зазор перед тестом)
- **Фолды:** 22 непересекающихся 60-дневных тестовых окна, с шагом 60 дней
- **Данные:** 435K разрешённых рынков, 470M торговых операций, сведённых в 82.6M временных агрегатов в parquet-файле с 30-дневными бакетами
- **Защита от временной утечки:** пересканирование 33 ГБ raw trades в 30-дневные частичные суммы, из которых позиции at-cutoff восстанавливаются SQL-фильтром `WHERE time_bucket <= T/bucket_size`

### Инфраструктура

Выбран DuckDB с потоковой обработкой и predicate pushdown на `time_bucket` для избежания загрузки 470M торговых операций в оперативную память. Bucket-подход даёт 225× ускорение (15 мин на фолд → 4 сек) и снижает пик памяти с 7.4 ГБ до 4.6 ГБ.

### Результаты Валидации

| Подмножество | Mean BSS | Median BSS | Fraction > 0 | 95% CI | p-value |
|---|---|---|---|---|---|
| All 22 folds | +0.196 | +0.159 | 22/22 (100%) | [+0.094, +0.297] | p = 2.38 × 10⁻⁷ (sign test) |
| Robust (folds 0–16, test ≥944) | +0.127 | +0.095 | 17/17 (100%) | — | — |

**О статистическом тесте.** Используется биномиальный sign test: при нулевой гипотезе «информированный консенсус не лучше сырого рынка» вероятность получить 22 положительных BSS из 22 независимых фолдов равна $p = 1/2^{22} \approx 2.38 \times 10^{-7}$. Это более консервативный выбор, чем t-test на массиве BSS: sign test не предполагает нормальности распределения BSS по фолдам и устойчив к выбросам. Код: [`scripts/eval_walk_forward.py:849-852`](https://github.com/Antopkin/delphi-press/blob/main/scripts/eval_walk_forward.py#L849-L852). 95% CI по mean BSS получен через block bootstrap (n=1000 resamples, block_size=3) в [`eval_walk_forward.py:833-846`](https://github.com/Antopkin/delphi-press/blob/main/scripts/eval_walk_forward.py#L833-L846), чтобы сохранить темпоральную корреляцию между соседними фолдами.

### Интерпретация

1. **100% положительных фолдов (22/22).** Информированный консенсус **всегда** улучшает сырую рыночную цену Polymarket, без исключений.

2. **Robust mean BSS = +0.127.** На подмножестве из 17 фолдов с надёжным объёмом тестовых рынков (≥ 944) улучшение Brier Score составляет 12.7% относительно сырых цен. **CI [+0.094, +0.297] рассчитан с использованием block bootstrap, сохраняющего темпоральную корреляцию между фолдами.**

3. **Инвертированная U-кривая.** BSS достигает пика при fold 9–11 (+0.21–+0.27) с оптимальным объёмом данных (~35K информированных бетторов), затем уменьшается по мере увеличения ликвидности рынка. Это соответствует литературе (Akey et al. 2025): информированные торговцы более значимы на тонких рынках.

4. **Стабильность яруса.** Метрика Jaccard overlap между множествами INFORMED бетторов в соседних фолдах = 0.613 (61%). Превышает пороговое значение 60%, подтверждая стабильность классификации яруса.

5. **Масштаб.** 1.74M профилированных кошельков, из них 348K в ярусе INFORMED — крупнейший published dataset бетторских профилей на Polymarket.

### Варианты без Extremizing

Абляция показала, что простая accuracy-weighted consensus с Bayesian shrinkage (k=15) оптимальна. Экстремизация (Satopää et al. 2014) фактически **вредит** (-64% BSS), вероятно, потому что информированные бетторы коррелированы (реагируют на те же рыночные сигналы), а не независимо информированы (частная информация).

### Корреляция новостей и рынка

Дополнительная валидация использует Spearman rank correlation и Granger causality test для проверки направления влияния новостного сигнала на цены:

- **Spearman ρ:** между абсолютным изменением цены $|\Delta p|$ и количеством релевантных новостей (требует min 5 datapoints)
- **Granger causality:** тестирует, предсказывает ли объём новостей изменения цены с лагом до 7 дней. ADF test проверяет стационарность; результат — (F-статистика, p-value, оптимальный лаг) или (None, None, None) при недостаточных данных

Эти метрики вычисляются в `src/eval/correlation.py` и используются для диагностики, но не входят в основную Brier Score оценку.

## Murphy Decomposition: Анализ компонент Brier Score

Чтобы глубже понять, откуда берётся ошибка предсказания, используется трёхкомпонентное разложение Brier Score, разработанное Мёрфи (1973). Оно разделяет BS на три ортогональные компоненты:

\[
BS = \text{REL} - \text{RES} + \text{UNC}
\]

где:

- **REL (надёжность)** — штраф за неправильную калибровку; показывает, насколько часто система выдаёт вероятность $p_k$, а фактическая база событий равна $\bar{o}_k \neq p_k$
- **RES (разрешающая способность)** — способность различать события; чем больше RES, тем лучше система предсказывает разные исходы
- **UNC (неопределённость)** — врождённая неопределённость в целевом наборе данных; зависит только от доли положительных примеров

### Формулы

Пусть $N$ — всего предсказаний, $o$ — вектор исходов, $p$ — вектор предсказанных вероятностей. Разделим предсказания на $K$ равноширинных бинов (по умолчанию $K = 10$):

**Надёжность (Reliability):**

\[
\text{REL} = \sum_{k=1}^{K} \frac{n_k}{N} \left(\bar{o}_k - \bar{p}_k\right)^2
\]

где $n_k$ — число предсказаний в бине $k$, $\bar{o}_k$ — среднее значение исходов в бине, $\bar{p}_k$ — среднее предсказанной вероятности в бине.

**Разрешающая способность (Resolution):**

\[
\text{RES} = \sum_{k=1}^{K} \frac{n_k}{N} \left(\bar{o}_k - \bar{o}\right)^2
\]

где $\bar{o}$ — общая доля положительных исходов (базовая частота).

**Неопределённость (Uncertainty):**

\[
\text{UNC} = \bar{o} \left(1 - \bar{o}\right)
\]

### Интерпретация

- **Высокая REL, низкая RES**: система хорошо откалибрована, но не различает события → предсказывает вероятность близко к 0.5 для всех случаев (бесполезно).
- **Низкая REL, высокая RES**: система хорошо различает события, но плохо откалибрована → отличные ранжирования, но вероятности смещены (исправимо через Platt scaling).
- **Высокая REL, высокая RES**: система хорошо различает события и правильно откалибрована → отличное качество.

### Пример

Для набора данных с 100 предсказаниями, где 30% положительных:

- $\bar{o} = 0.3$
- $\text{UNC} = 0.3 \times 0.7 = 0.21$ (фиксировано для набора)
- Если система предсказала BS = 0.15:
  - Вариант A: REL = 0.02, RES = 0.08 → $0.02 - 0.08 + 0.21 = 0.15$ ✓ (хорошо откалибрована, хорошо различает)
  - Вариант B: REL = 0.15, RES = 0.01 → $0.15 - 0.01 + 0.21 = 0.35$ ✗ (плохо откалибрована)

### Применение в Delphi Press

Разложение вычисляется функцией `brier_decomposition()` с параметром `n_bins=10` (равноширинные бины). Результат включается в еженедельный мониторинг:

```python
from src.eval.metrics import brier_decomposition

decomp = brier_decomposition(pred_probs, outcomes, n_bins=10)
print(f"REL={decomp.reliability:.4f}, RES={decomp.resolution:.4f}, UNC={decomp.uncertainty:.4f}")

if decomp.reliability > 0.05:
    # Применить Platt scaling (калибровка)
    pass
if decomp.resolution < 0.10:
    # Система недостаточно различает → пересмотреть агентов
    pass
```

## Калибровка: Slope и Expected Calibration Error

Помимо общего Brier Score, система отслеживает два специфичных показателя калибровки для диагностики и улучшения.

### Calibration Slope (OLS регрессия)

**Определение:** OLS-регрессия логистическая: $\text{outcome} \sim \text{predicted\_probability}$. Получается наклон (slope).

\[
\text{slope} = \frac{\text{cov}(p, o)}{\text{var}(p)}
\]

где $p$ — вектор предсказанных вероятностей, $o$ — вектор исходов.

**Интерпретация:**

- **slope ≈ 1.0** — идеальная калибровка (система соответствует действительности)
- **slope < 1.0** — переуверенность (predictions слишком экстремальны: слишком много ~0 и ~1, мало средних значений)
- **slope > 1.0** — недоуверенность (predictions слишком близко к 0.5, не достаточно экстремальны)

**Пример:** если slope = 0.8, система выдала среднюю вероятность $\bar{p} = 0.50$, а реальная доля положительных $\bar{o} = 0.30$, то система переуверена на 20%.

**Исправление:** можно применить линейную трансформацию (Platt scaling):

\[
p_{\text{calibrated}} = \text{sigmoid}(a \cdot \text{logit}(p) + b)
\]

где $a, b$ подбираются на валидационном наборе.

### Expected Calibration Error (ECE)

**Определение:** ECE измеряет среднее абсолютное расхождение между предсказанной вероятностью и фактической частотой в каждом бине.

\[
\text{ECE} = \sum_{k=1}^{K} \frac{n_k}{N} \left| \bar{p}_k - \bar{o}_k \right|
\]

**Алгоритм (equal-frequency binning):**

1. Сортировать предсказания по возрастанию вероятности
2. Разделить на $K$ бинов равного размера (не равной ширины)
3. Для каждого бина вычислить среднюю предсказанную $\bar{p}_k$ и среднюю реальную $\bar{o}_k$
4. Суммировать взвешенные абсолютные разности

**Интерпретация:**

- **ECE < 0.05** — отличная калибровка
- **0.05 ≤ ECE < 0.10** — хорошая калибровка
- **ECE ≥ 0.10** — плохая калибровка

**Пример:** если ECE = 0.08, в среднем предсказанная вероятность отличается от реальной частоты на 8 процентных пункта.

### Применение в Delphi Press

```python
from src.eval.metrics import calibration_slope, expected_calibration_error

slope = calibration_slope(pred_probs, outcomes)
ece = expected_calibration_error(pred_probs, outcomes, n_bins=10)

if slope < 0.9:
    logger.warning(f"System is overconfident: slope={slope:.3f}")
    # Consider Platt scaling

if ece > 0.10:
    logger.warning(f"Poor calibration: ECE={ece:.4f}")
    # Consider retraining or adjusting thresholds
```

## Источники Ground Truth

Для ретроспективного тестирования система использует многоуровневый подход к восстановлению фактических заголовков, опубликованных в целевых изданиях за прошлые даты.

### Wayback Machine CDX API

**Назначение:** первичный источник ground truth для всех исторических снимков RSS-потоков. Обеспечивает точное восстановление заголовков, которые были видны пользователям в целевую дату.

**Архитектура запроса:**

Система использует двухэтапный процесс через API Wayback Machine:

**Этап 1: CDX Search (поиск снимков)**

```
GET https://web.archive.org/cdx/search/cdx
  ?url={rss_url}
  &output=json
  &from={YYYYMMDDHHMMSS}
  &to={YYYYMMDDHHMMSS}
  &statuscode=200
  &limit=5
  &fl=timestamp,original
```

Параметры:

- `url` — URL RSS-потока (напр., `https://tass.ru/rss`)
- `from/to` — диапазон поиска (обычно window_hours=24, по умолчанию)
- `statuscode=200` — только успешные HTTP-снимки
- `limit=5` — максимум 5 снимков за дату (достаточно для покрытия)
- `fl=timestamp,original` — возвращать время и исходный URL

**Ответ CDX (JSON):**

```json
[
  ["timestamp", "original"],
  ["20260405000000", "https://tass.ru/rss"],
  ["20260405060000", "https://tass.ru/rss"],
  ["20260405120000", "https://tass.ru/rss"]
]
```

Первая строка — заголовки, остальные — снимки с временными метками.

**Этап 2: RSS parsing (извлечение заголовков)**

Для каждого снимка выполняется запрос:

```
GET https://web.archive.org/web/{timestamp}/{original_rss_url}
```

Ответ содержит XML RSS-ленту с элементами `<item><title>...`.

Функция `_extract_titles_from_rss(xml_text)` парсит XML и извлекает все `<title>` из блоков `<item>`:

```python
def _extract_titles_from_rss(xml_text: str) -> list[str]:
    root = ET.fromstring(xml_text)
    titles = []
    for item in root.iter("item"):
        title_el = item.find("title")
        if title_el is not None and title_el.text:
            titles.append(title_el.text.strip())
    return titles
```

**Дополнительные меры:**

- **Задержка между запросами:** 1 секунда между снимками (вежливое использование CDX)
- **Дедупликация:** заголовки сортируются и удаляются дубликаты, сохраняется исходный порядок
- **Обработка ошибок:** все исключения (httpx, XML parsing) логируются; функция возвращает пустой список, если Wayback недоступна

**Примеры источников с хорошим покрытием на Wayback:**

### RSS-снимки (текущие архивы проекта)

**Назначение:** локальный источник, дополняющий Wayback для свежих данных и ускорения оценки.

Система Delphi Press собирает RSS-записи в реальном времени из 16 источников в таблицу `raw_articles` SQLite базы данных. Каждая запись содержит:

### Разрешение исходов: как определяется $o_i$ (0 или 1)

После сбора ground truth (списка фактических заголовков за целевую дату) система определяет бинарный исход $o_i$ для каждого предсказания. Процесс основан на **TopicMatch** — степени совпадения темы предсказания с реальностью.

**Определение исхода:**

\[
o_i = \begin{cases}
1.0, & \text{если } \text{TopicMatch} > 0 \text{ (есть хотя бы частичное совпадение)} \\
0.0, & \text{если } \text{TopicMatch} = 0 \text{ (полный промах)}
\end{cases}
\]

**Граничные случаи и разрешение конфликтов:**

1. **Несколько совпадений одного предсказания**
   - Если предсказание близко к двум реальным заголовкам, берётся **максимум** TopicMatch
   - Пример: предсказание "Путин встретился с главой МИД" совпадает с двумя реальными: F1 = 0.85 (полное совпадение) и F1 = 0.62 (частичное) → $o_i = 1.0$ (максимум)

2. **Нет ground truth на целевую дату**
   - Если Wayback и RSS оба пусты, дата исключается из оценки (не учитывается в $N$)
   - Логируется warning

3. **BERTScore пограничный случай (0.55 ≤ F1 < 0.60)**
   - Направляется на LLM-arbiter
   - LLM выносит суждение (decision: "ДА" или "НЕТ")
   - TopicMatch = 1.0 если decision="ДА" и confidence ≥ 0.80, иначе 0.0 (консервативно)

4. **Конфликты между источниками (RSS проекта vs Wayback)**
   - Если RSS проекта содержит заголовок H1, а Wayback — H2, и оба совпадают с предсказанием (но с разной F1):
   - Берётся заголовок с наивысшим BERTScore F1
   - Логируется конфликт для последующего аудита

5. **Ошибки парсинга XML**
   - Снимок Wayback повреждён → пропускается, продолжаются остальные снимки из окна
   - Если все снимки повреждены → дата помечается как "unresolved"

**Функции разрешения в коде:**

```python
# Ground truth collection (src/eval/ground_truth.py)
headlines = await fetch_headlines_from_wayback(rss_url, target_date)

# Matching (src/eval/bertscore_eval.py) — функция на стадии рефакторинга
best_f1 = max([bertscore_f1(prediction, h) for h in headlines])

if best_f1 >= 0.78:
    o_i = 1.0  # TopicMatch = 1.0
elif best_f1 >= 0.60:
    o_i = 1.0  # TopicMatch = 0.5, но o_i всё равно = 1.0
elif best_f1 >= 0.55:
    llm_decision = await llm_arbiter(prediction, best_matching_headline)
    o_i = 1.0 if llm_decision.decision == "ДА" else 0.0
else:
    o_i = 0.0  # TopicMatch = 0.0
```

**Статистика разрешения (пилотные данные):**

| Сценарий | Доля случаев | Исход |
|---|---|---|
| F1 ≥ 0.78 (полное совпадение) | ~35% | $o_i = 1.0$, без LLM |
| 0.60 ≤ F1 < 0.78 (частичное) | ~45% | $o_i = 1.0$, без LLM |
| 0.55 ≤ F1 < 0.60 (пограничный) | ~12% | LLM-арбитр (~75% → 1.0) |
| F1 < 0.55 (промах) | ~8% | $o_i = 0.0$, без LLM |

## BERTScore: Семантическое сходство заголовков

### Что измеряет

BERTScore вычисляет семантическую близость между двумя текстами через контекстные эмбеддинги BERT, избегая ограничений простых метрик на основе n-грамм (ROUGE, BLEU). Для текста $T_1$ и $T_2$:

$$\text{BERTScore F1} = \frac{2 \times P \times R}{P + R}$$

где:

- **Precision:** доля токенов $T_1$, имеющих высокое косинусное сходство с токеном в $T_2$
- **Recall:** доля токенов $T_2$, совпадающих с токеном в $T_1$

**Интуиция:** для предсказанного заголовка "Путин обсудил экономику с Минфином" и реального "Премьер-министр провёл переговоры по финансам", ROUGE/BLEU дадут низкий скор (нет общих слов), а BERTScore распознаёт семантическое совпадение.

### Использование в TopicMatch

BERTScore встроен как второй этап трёхступенчатого алгоритма TopicMatch:

**Шаг 2: BERTScore (~50 мс на батч, локально)**
```python
# После keyword screening (Шаг 1) отобраны кандидаты
# Теперь вычисляем BERTScore F1 против каждого кандидата
candidates = ["Путин встретился с главой Минфина", "Экономика растёт на 2%", ...]
predicted = "Путин обсудил финансовую политику"

f1_scores = bertscore_f1(predicted, candidates, lang="ru")
# f1_scores = [0.82, 0.41, ...]

best_f1 = max(f1_scores)  # 0.82

if best_f1 >= 0.78:
    return 1.0  # Полное совпадение (TopicMatch = 1.0)
elif best_f1 >= 0.60:
    return 0.5  # Верная тема, неверный исход (TopicMatch = 0.5)
else:
    # Пригодится LLM-арбитр для пограничных случаев
    ...
```

### Калибровка пороговых значений

Пороги F1 (0.78, 0.60, 0.55) определены по литературе на задачах перевода и суммаризации, но **требуют локальной калибровки на русскоязычных заголовках** (5–15 слов).

**Протокол калибровки (20–30 примеров):**

1. Вручную отобрать 20–30 пар: (предсказанный_заголовок, реальный_заголовок).
2. Для каждой пары назначить эталонный TopicMatch вручную:
   - 1.0: полное совпадение (одно и то же событие, сопоставимая позиция)
   - 0.5: верная тема, но другой угол или неточный исход
   - 0.0: промах (другая тема)

3. Вычислить BERTScore F1 для каждой пары локально.
4. Построить ROC-кривую и найти оптимальные пороги (максимизирующие F1 метрику классификации).

**Ожидаемый результат:** пороги сместятся на 5–10% от опубликованных значений, но направление останется тем же (0.78 > 0.60 > 0.55).

!!! warning "До реализации"
    Пороги в коде пока жёстко закодированы. После первого пилота (N=50) требуется провести калибровку и обновить в `src/eval/bertscore_eval.py`.

### Модели по языкам

| Язык | Модель | Качество | Использование |
|------|--------|----------|----------------|
| Русский (монолингвальный) | `DeepPavlov/rubert-base-cased` | Лучшее для русских заголовков | Основное (RU → RU) |
| Английский | `roberta-large` | Стандарт авторов BERTScore | Основное (EN → EN) |
| Кроссъязычный (RU ↔ EN) | `xlm-roberta-base` | Хорошее для перевода | ТАСС EN vs. исходные RU |

## LLM-as-Judge: Оценка пограничных случаев

### Когда инвокируется

LLM-судья вызывается только для пограничных случаев BERTScore, когда автоматическая оценка недостаточна:

$$0.55 \leq \text{F1} < 0.60 \implies \text{LLM-arbiter}$$

Это охватывает ~10% прогнозов и экономит ~90% LLM-вызовов на оценку TopicMatch. Дополнительно, LLM оценивает StyleMatch всегда (как отдельный компонент CompositeScore).

### Структура промпта для TopicMatch

Когда BERTScore F1 попадает в пограничный диапазон, система просит LLM-судью разрешить неоднозначность:

```
System:
Ты опытный редактор новостей. Твоя задача — определить,
является ли предсказанный заголовок корректным прогнозом
фактически опубликованного заголовка.

User:
Предсказанный заголовок:
"Путин провел переговоры с главой МИД о дипломатии"

Фактически опубликованный заголовок:
"Российский лидер встретился с министром иностранных дел"

Контекст события:
- Дата: 2026-04-05
- Издание: ТАСС
- Горизонт: 3 дня

Вопрос: Прогноз ли это верного события?
Ответь одним словом: ДА или НЕТ.
Объясни вкратце (1 предложение) логику.

Expected output (JSON):
{
  "decision": "ДА",
  "confidence": 0.92,
  "explanation": "Одно и то же событие: встреча Putin с главой МИД."
}
```

**Вывод LLM переводится в TopicMatch:**

- confidence ≥ 0.80 и decision == "ДА" → TopicMatch = 1.0
- confidence < 0.80 или decision == "НЕТ" → TopicMatch = 0.0

### Структура промпта для StyleMatch

Отдельный промпт для оценки стилевого соответствия заголовка (компонент CompositeScore):

```
System:
Ты корректор, специалист по редакционному стилю.
Оцени, насколько предсказанный заголовок соответствует
стилю типичных заголовков издания.

User:
Издание: BBC News Russian
Стилевой эталон (примеры типичных заголовков):
1. "В Москве прошла акция протеста против повышения тарифов"
2. "Эксперты: инфляция замедляется из-за снижения спроса"
3. "Миллионы избирателей проголосовали на выборах главы"

Предсказанный заголовок:
"Путин совещается по финансово-хозяйственным вопросам"

Критерии оценки:
1. Длина (слов): 5–15 — оптимально для издания
2. Лексика: соответствует ли эталонам по сложности и формальности
3. Структура: SVO (Subject-Verb-Object) или инверсия
4. Тональность: нейтральная, информативная (не сенсационная)

Оцени по шкале 1–5, где:
1 = совсем не соответствует
3 = частично соответствует
5 = идеально соответствует

Answer JSON:
{
  "score": 4,
  "length_ok": true,
  "lexicon_match": "formal, подходит",
  "structure_match": "SVO, стандартная",
  "tone_match": "neutral, информативный",
  "explanation": "Хороший заголовок, слегка многословен."
}
```

**Вывод нормализуется на [0,1]:** `StyleMatch = score / 5`

### Ограничения LLM-as-Judge

1. **Self-Enhancement Bias:** Если одна и та же модель (Claude) генерирует заголовок и оценивает его, возможен systematic upward bias. **Решение:** использовать разные модели или явно указать в system prompt, что судья оценивает работу конкурента, не свою.

2. **Position Bias:** Первый вариант часто получает более высокий скор. **Решение:** рандомизировать порядок заголовков при оценке.

3. **Verbosity Bias:** LLM предпочитает более длинные и подробные ответы. **Решение:** явно указать пределы длины в эталонных примерах.

4. **Стоимость:** При N=350 пар × 500 токенов = 175K токенов input → ~$0.53 (Claude Sonnet 4.6). При N=1400 → ~$2.10. Умеренно дорого, но оправдано высокой точностью.

5. **Глюки на граничных случаях:** Редко LLM может ошибиться и дать confidence = 0.50–0.70, что усложняет интерпретацию. **Решение:** установить пороговое значение confidence ≥ 0.70 или < 0.30 для принятия решения; если в диапазоне [0.30, 0.70], дополнительный вызов или переход на более консервативное решение (0.5 вместо 1.0).

## Цели качества и Production Targets

### Для v1.0 (Уровень Prediction Market)

**Brier Score:**

- **Цель:** BS < 0.20
- **Обоснование:** уровень Metaculus-сообщества (BS = 0.182) и Polymarket (BS ≈ 0.187). Реалистично для многоагентной системы без человеческой экспертизы.
- **Тест:** пилот N=50 с 95% CI ±0.03–0.05; если BS = 0.205 ± 0.04, цель достигнута.

**Brier Skill Score:**

- **Цель:** BSS > 0.20
- **Интерпретация:** система превосходит случайное угадывание минимум на 20%.

**CompositeScore:**

- **TopicMatch:** > 0.60 для > 80% предсказаний
  - Означает, что в 80% случаев система правильно определяет, будет ли тема в прессе
  - Если TopicMatch < 0.60 в 30% случаев → потребуется улучшение EventTrendAnalyzer
- **SemanticSim:** F1 (BERTScore) > 0.65 в среднем
  - Заголовки логичны и близки к фактическим по смыслу
- **StyleMatch:** > 0.50 в среднем
  - Стиль соответствует профилю издания

### Для v2.0 (Уровень обученных прогнозистов)

**Brier Score:**

- **Цель:** BS < 0.12
- **Обоснование:** уровень Good Judgment Project top forecasters (BS ≈ 0.14–0.15) и GPT-4.5 (BS = 0.101).
- **Timeline:** 6–12 месяцев после v1.0, требуется:
  - Тонкая калибровка весов персон на основе накопленной BS-статистики
  - Расширение OutletProfile на 50+ примеров per outlet
  - Возможное дополнение инструментом anomaly detection для breaking news

**BSS:**

- **Цель:** BSS > 0.52
- **Интерпретация:** система превосходит случайное угадывание более чем на половину максимально возможного улучшения.

### Мониторинг в Production

**Ежедневные метрики (на скользящем окне 14 дней):**

- Mean BS, Min BS, Max BS (следить за дрейфом)
- Fraction TopicMatch > 0 (процент полных совпадений)
- Per-outlet CompositeScore (специфичность по изданиям)

**Еженедельные метрики (за последние 7 дней):**

- Decomposition BS = Reliability − Resolution + Uncertainty
  - Если Reliability > 0.05 → применить Platt scaling
  - Если Resolution < 0.10 → система недостаточно дискриминативна

**Ежемесячные метрики (за весь месяц):**

- Обновление весов персон в `judge.py` на основе per-persona BS
- Проверка per-outlet StyleMatch; если < 0.45 → пересмотреть OutletProfile

**Критические пороги (требуют срочного реагирования):**

- BS > 0.22 (v1.0) или BS > 0.14 (v2.0) → разбор, фикс процесса
- TopicMatch.mean < 0.45 → проблема в EventTrendAnalyzer или EventFilter
- StyleMatch.mean < 0.40 → проблема в StyleReplicator или OutletProfile
