# Раунды Дельфи: R1 и R2

В пайплайне Delphi Press используется двухраундовый метод Дельфи (Dalkey & Helmer 1963) с пятью синтетическими экспертными персонами и алгоритмической медиацией между раундами. Каждый раунд порождает независимые вероятностные оценки, которые затем агрегируются в финальный прогноз.

## Экспертные персоны

Пять персон сформированы по профилю когнитивных смещений из литературы по прогнозированию (Tetlock 2005, Taleb 2008). Каждая персона имеет уникальный системный промпт, веса и диапазон температур LLM.

| Персона | ID | Вес R1 | Переоценивает | Недооценивает | Якорь | T°R1 | T°R2 |
|---------|----|----|-------------|------------|----|----|-----|
| **Реалист** | `realist` | 0.22 | инерция статус-кво, предсказуемость бюрократии | чёрные лебеди, скорость эскалации, роль личных решений лидеров | исторические прецеденты | 0.7 | 0.6 |
| **Геостратег** | `geostrateg` | 0.20 | рациональность государств, значимость геополитических факторов | внутриполитические случайности, роль технологий | модель великодержавного соперничества | 0.7 | 0.6 |
| **Экономист** | `economist` | 0.20 | экономическая рациональность, рыночные индикаторы | идеологические и эмоциональные факторы, случайности | экономический календарь и рыночный консенсус | 0.7 | 0.6 |
| **Медиа-эксперт** | `media_expert` | 0.18 | медиа-циклы, предсказуемость редакционных решений | технические/процедурные события, медленные кризисы | текущий новостной цикл | 0.7 | 0.6 |
| **Контрариан** | `devils_advocate` | 0.20 | чёрные лебеди, хрупкость систем | устойчивость статус-кво, вероятность скучных исходов | маловероятные высокоимпактные сценарии | **0.9** | 0.6 |

### Примечания о температурах

- **R1 Реалист / Геостратег / Экономист / Медиа-эксперт**: температура **0.7** (сбалансированное исследование)
- **R1 Контрариан**: температура **0.9** (повышенная стохастичность для поиска альтернативных сценариев)
- **Медиатор**: температура **0.5** (консервативная, сосредоточена на синтезе)
- **R2 все персоны**: температура **0.6** (уменьшена для сведения к истинному консенсусу)
- **max_tokens**: не ограничен (неограниченный выход)

Исходный код: [`src/llm/router.py`](https://github.com/antopkin/delphi_press/blob/main/src/llm/router.py) (строки 98–173).

## Раунд 1: Независимая оценка

В Раунде 1 каждая персона независимо анализирует траектории событий и порождает список вероятностных прогнозов. Персоны не видят друг друга и оцениваются параллельно. Результат R1 — пять независимых `PersonaAssessment`.

### PersonaAssessment (R1 выход)

```python
PersonaAssessment(
    persona_id: str,           # "realist", "geostrateg", etc.
    round_number: int,         # 1
    predictions: list[PredictionItem],  # 5–15 прогнозов
    cross_impacts_noted: list[str],
    blind_spots: list[str],
    confidence_self_assessment: float,  # 0.0–1.0
    revisions_made: list[str],          # пусто в R1
    revision_rationale: str,            # пусто в R1
)
```

### PredictionItem (единичный прогноз)

Каждый `PredictionItem` описывает одно ожидаемое событие:

| Поле | Тип | Описание |
|------|-----|---------|
| `event_thread_id` | `str` | ID EventThread, к которому относится прогноз |
| `prediction` | `str` | Конкретное утверждение (не общие слова) |
| `probability` | `float` | 0.0–1.0: оценка вероятности события |
| `newsworthiness` | `float` | 0.0–1.0: вероятность попасть в выпуск (фильтр гейткипинга) |
| `scenario_type` | `ScenarioType` | `BASELINE` / `OPTIMISTIC` / `PESSIMISTIC` / `BLACK_SWAN` / `WILDCARD` |
| `reasoning` | `str` | 3–7 предложений с цепочкой рассуждений |
| `key_assumptions` | `list[str]` | 2–4 ключевых предпосылки |
| `evidence` | `list[str]` | Ссылки на конкретные факты из контекста |
| `conditional_on` | `list[str]` | ID других PredictionItem, от которых зависит этот |
| **predicted_date** | `date \| None` | Прогнозируемая дата события (YYYY-MM-DD). None = target_date контекста |
| **uncertainty_days** | `float` | Неопределённость ±N дней (0.5 = высокая точность, 7.0 = тренд) |
| **causal_dependencies** | `list[str]` | event_thread_id событий, от которых каузально зависит этот прогноз |
| **confidence_interval_95** | `(float, float) \| None` | 95% доверительный интервал для probability (Barrett et al., RAND 2025) |

Исходный код: [`src/schemas/agent.py:110–151`](https://github.com/antopkin/delphi_press/blob/main/src/schemas/agent.py).

## Медиатор: Синтез расхождений

После R1 **медиатор** не агрегирует вероятности (это делает Judge). Вместо этого медиатор:

1. Классифицирует события на три категории
2. Анонимизирует позиции экспертов
3. Порождает ключевые вопросы для разрешения спорных ситуаций
4. Передаёт синтез в R2

### Алгоритм анонимизации

Анонимизация детерминистична: все методы используют общую таблицу отображения `persona_id` → `Expert A, B, C...`

```python
def _build_label_map(assessments: list[PersonaAssessment]) -> dict[str, str]:
    """Create stable persona_id → 'Expert A/B/C...' mapping."""
    labels = [f"Expert {c}" for c in string.ascii_uppercase[: len(assessments)]]
    seed = hash(tuple(sorted(a.persona_id for a in assessments)))  # ← ДЕТЕРМИНИСТИЧНЫЙ
    rng = random.Random(seed)
    rng.shuffle(labels)
    return {a.persona_id: label for label, a in zip(labels, assessments)}
```

Ключевая особенность: **отображение стабильно во всех методах медиатора, потому что используется один и тот же seed на основе отсортированных persona_id**. Если одна персона не участвует, меняется seed и переперемешивается вся таблица. Это скрывает порядок персон, но сохраняет воспроизводимость.

Исходный код: [`src/agents/forecasters/mediator.py:181–206`](https://github.com/antopkin/delphi_press/blob/main/src/agents/forecasters/mediator.py).

### Классификация событий

Для каждого события медиатор вычисляет:

```python
spread = max(probabilities) - min(probabilities)
median_prob = median(probabilities)
```

Затем классифицирует:

| Категория | Условие | Действие |
|-----------|---------|---------|
| **Консенсус** (ConsensusArea) | spread < 0.15 и ≥3 персон упомянули | Переносится в R2 без изменений |
| **Спор** (DisputeArea) | spread ≥ 0.15 и ≥3 персон упомянули | Генерируется ключевой вопрос для R2 |
| **Пробел** (GapArea) | < 3 персон упомянули событие | Помечается как упущение |

### MediatorSynthesis (выход медиатора)

```python
MediatorSynthesis(
    consensus_areas: list[ConsensusArea],
    disputes: list[DisputeArea],
    gaps: list[GapArea],
    cross_impact_flags: list[CrossImpactFlag],
    overall_summary: str,
    supplementary_facts: list[str],
)
```

```python

```

## Раунд 2: Пересмотр после медиации

В Раунде 2 **медиатор запускается первым** (последовательно), синтез сохраняется в контекст, затем **все 5 персон запускаются параллельно** с `MediatorSynthesis` в контексте.

### Поток R2 в оркестраторе

```python
async def _run_delphi_r2(stage_def, context, start_ns):
    # Фаза 1: Медиатор (последовательно)
    mediator_result = await mediator.run(context)
    context.merge_agent_result(mediator_result)  # ← MediatorSynthesis попадает в context

    # Фаза 2: 5 персон параллельно (видят context.mediator_synthesis)
    r2_results = await asyncio.gather(
        realist.run(context),
        geostrateg.run(context),
        economist.run(context),
        media_expert.run(context),
        devils_advocate.run(context),
    )
```

Исходный код: [`src/agents/orchestrator.py:286–366`](https://github.com/antopkin/delphi_press/blob/main/src/agents/orchestrator.py).

### Какие данные видят персоны в R2?

Каждая персона в R2 получает:

1. **Исходные данные** (как в R1):
   - Траектории событий
   - Матрица перекрёстных влияний
   - Дата целевого события
   - Условия горизонта (дней до события, категория спешности)

2. **MediatorSynthesis** (новое):
   - Области консенсуса (события, на которые все согласны)
   - Области споров (события с расхождениями; **без раскрытия persona_id**, только "Expert A, B, C...")
   - Пробелы (события, упущенные большинством)
   - Ключевые вопросы (сгенерированные медиатором)
   - Перекрёстные флаги (зависимости от спорных событий)

### PersonaAssessment (R2 выход)

R2 выходы структурированы почти как R1, но с дополнительными полями:

```python
PersonaAssessment(
    persona_id: str,           # "realist", "geostrateg", etc.
    round_number: int,         # 2
    predictions: list[PredictionItem],  # может отличаться от R1
    cross_impacts_noted: list[str],
    blind_spots: list[str],
    confidence_self_assessment: float,
    revisions_made: list[str],          # ← НОВОЕ: что изменилось от R1?
    revision_rationale: str,            # ← НОВОЕ: почему изменилось?
)
```

### Стратегия пересмотра

Персоны следуют одной из трёх стратегий:

1. **Конвергенция** (convergence): персона видит консенсус и адаптирует свою оценку ближе к медиане
2. **Дивергенция** (divergence): персона видит спор, но упорствует в своём анализе с обоснованием ("я несогласен, потому что...")
3. **Поиск дополнительных доказательств** (epistemic move): персона пересмотрит прогноз только если медиатор поднял ранее упущенный фактор

Температуры R2 снижены до 0.6 для **всех персон** (включая контрариана), чтобы поощрить схождение к истинному консенсусу, не потеряв разнообразия аргументов.

## Judge: Агрегация и калибровка (Stage 6)

После обоих раундов **Judge** (от лат. _судья_) агрегирует оценки персон, применяет рыночные сигналы, и выдаёт финальный ранжированный список прогнозов.

### Двухшаговый алгоритм Judge

Judge выполняет две последовательные стадии:

1. **Step 6a: Event-level aggregation** — агрегация вероятностей персон в события (выход: `PredictedTimeline`)
2. **Step 6b: Headline selection** — отбор топ-N событий для финального списка (выход: `list[RankedPrediction]`)

Исходный код: [`src/agents/forecasters/judge.py:101–646`](https://github.com/antopkin/delphi_press/blob/main/src/agents/forecasters/judge.py).

## Формула рыночного сигнала (Market Weight)

Judge интегрирует рыночные сигналы из Polymarket (Phase 4) как "виртуальную 6-ю персону" с динамически вычисляемым весом.

### Компоненты рыночного веса

Вес рынка вычисляется по формуле:

$$w_{\text{market}} = w_0 \times f_{\text{liq}} \times f_{\text{vol}} \times r_{\text{dist}} \times f_{\text{horizon}}$$

Где:

| Компонента | Значение/Формула | Описание |
|-----------|---------|----------|
| $w_0$ (базовый вес) | 0.15 | Base market weight |
| $f_{\text{liq}}$ (ликвидность) | $\max(0.5, \min(\log_{10}(\text{liq})/6, 1.5))$ | Логарифмический множитель на основе объёма в USD; пол 0.5, потолок 1.5 |
| $f_{\text{vol}}$ (волатильность) | $\frac{1}{1 + \sigma_{7d}}$ | Гиперболический дисконт; $\sigma_{7d}$ — волатильность за 7 дней |
| $r_{\text{dist}}$ (надёжность) | 1.0 или 0.0 | Флаг: распределение вероятности надёжно (true) или нет (false) |
| $f_{\text{horizon}}$ (горизонт) | 1.0, 0.75, или 0.5 | 7-дневная метрика более значима для близких рынков; см. таблицу ниже |

### Таблица коэффициентов горизонта

| Дни до истечения | $f_{\text{horizon}}$ | Обоснование |
|---|---|---|
| ≤ 90 дней | 1.0 | Метрики 7d точны и своевременны |
| 90–180 дней | 0.75 | Метрики становятся менее релевантны |
| > 180 дней | 0.5 | Долгосрочные рынки игнорируют недавние движения |

### Пороги включения рынка

Рынок включается в агрегацию только если:

| Параметр | Пороговое значение | Описание |
|----------|---------|----------|
| `MARKET_MIN_LIQUIDITY` | \$10,000 | Минимальный объём для надёжности сигнала |
| `MARKET_MIN_PROBABILITY` | 0.10 | Вероятность не может быть ниже 10% |
| `MARKET_MATCH_TIER1_SCORE` | 0.65 | Пороговая оценка для прямого совпадения (см. раздел "Fuzzy matching") |

## Платтова калибровка (Platt Scaling)

После вычисления взвешенной медианы Judge применяет **Platt scaling** для экстремизации вероятностей:

$$p_{\text{calibrated}} = \sigma(a \cdot \text{logit}(p_{\text{raw}}) + b)$$

Где:

- $\text{logit}(p) = \log\left(\frac{p}{1-p}\right)$ — логит-преобразование
- $\sigma(x) = \frac{1}{1 + e^{-x}}$ — логистическая функция
- $a$ и $b$ — калибровочные параметры

### Значения параметров

| Параметр | Значение | Источник |
|----------|---------|----------|
| $a$ (extremization) | 1.5 | docs/05-delphi-pipeline.md §5.1 |
| $b$ (bias) | 0.0 | docs/05-delphi-pipeline.md §5.1 |

### Эффект экстремизации

- $a > 1$ растягивает вероятности, отодвигая их от 0.5 (лучше различает уверенные прогнозы)
- $a = 1.5$ означает, что вероятность 0.7 преобразуется в ≈ 0.74, а 0.3 в ≈ 0.24
- $b = 0$ означает отсутствие смещения (симметричная калибровка)

### Границы зажима

Перед калибровкой вероятность зажимается в диапазон [0.01, 0.99] для численной стабильности:

$$p' = \max(0.01, \min(0.99, p_{\text{raw}}))$$

## Штраф за неопределённость

Когда консенсус отсутствует (события спорны), Judge применяет **штраф за неопределённость**:

$$p_{\text{with\_penalty}} = p_{\text{raw}} \times k_{\text{penalty}}$$

Где:

| Параметр | Значение | Условие |
|----------|---------|---------|
| $k_{\text{penalty}}$ | 0.80 | Если spread ≥ 0.30 (CONTESTED уровень) |
| Иначе | 1.0 | Если spread < 0.30 (CONSENSUS или MAJORITY_WITH_DISSENT) |

### Классификация согласия

| Уровень | Условие | spread |
|---------|---------|---------|
| **CONSENSUS** | spread < 0.15 | Все 5 персон очень близки |
| **MAJORITY_WITH_DISSENT** | 0.15 ≤ spread < 0.30 | Большинство согласны, есть меньшинство |
| **CONTESTED** | spread ≥ 0.30 | Значительные разногласия; штраф применяется |

## Взвешенная медиана персон

Judge вычисляет взвешенную медиану вероятностей персон:

$$p_{\text{raw}} = \text{median}_{\text{weighted}}(p_1, p_2, \ldots, p_5; w_1, w_2, \ldots, w_5)$$

Алгоритм:

1. Отсортировать пары (вероятность, вес) по возрастанию вероятности
2. Накопить веса слева направо
3. Вернуть вероятность, где накопленный вес впервые превышает 50% от суммы всех весов

## Horizon-адаптивные веса (Adaptive Persona Weights)

Judge применяет горизонт-специфичные корректировки к базовым весам персон. Исследование показало (arXiv 2511.18394, GJP data), что разные персоны эффективнее для разных горизонтов:

$$w_{\text{final}}(p) = \max(0.01, w_{\text{base}}(p) + \delta(p, h))$$

Где:
- $h$ — горизонт-бэнд (immediate, near, medium)
- $\delta(p, h)$ — горизонт-корректировка из таблицы ниже
- $\max(0.01, \ldots)$ — минимальный вес не может быть менее 0.01

### Таблица горизонт-коррекций

| Горизонт | Период | Реалист | Геостратег | Экономист | Медиа-эксперт | Контрариан |
|----------|--------|---------|-----------|-----------|-------------|-----------|
| **immediate** | 1–2 дня | −0.02 | −0.02 | +0.02 | +0.03 | −0.01 |
| **near** | 3–4 дня | 0.00 | 0.00 | 0.00 | 0.00 | +0.02 |
| **medium** | 5+ дней | +0.03 | +0.02 | −0.01 | −0.02 | −0.02 |

### Пример расчёта

Для события в горизонте "near" (4 дня):

- Реалист: \(w = 0.22 + 0.00 = 0.22\)
- Контрариан: \(w = 0.20 + 0.02 = 0.22\)
- Экономист: \(w = 0.20 + 0.00 = 0.20\)
- (остальные не меняются)

Затем веса нормализуются так, чтобы их сумма = 1.0.

### Обоснование коррекций

- **immediate**: Медиа-эксперт лучше (новостной цикл, расписания), Реалист хуже (исторические прецеденты менее актуальны)
- **near**: Контрариан более ценен (максимальная зона неопределённости, по Tetlock/GJP)
- **medium**: Реалист + Геостратег лучше (базовые ставки, структурные факторы доминируют)

## Fuzzy Matching рынков к событиям

Judge должен сопоставить события из пайплайна с рынками из Polymarket. Для этого используется **трёхуровневая нечёткая подстановка** (three-tier fuzzy matching):

### Алгоритм

Входные данные:
- `search_texts` — список текстов события (event_thread_id, тексты предсказаний)
- `market_index` — словарь {lowercase_title → market_dict} для всех доступных рынков
- `event_categories` — категории события (опционально, для Tier 2)

Процесс:

1. **Вычислить оценку сходства для всех рынков** используя `token_sort_ratio` из rapidfuzz:

$$\text{score} = \frac{\text{token_sort_ratio}(\text{search_text}, \text{market_title})}{100}$$

Функция `token_sort_ratio`:
- Разбить обе строки на токены (слова)
- Отсортировать токены
- Вычислить коэффициент Левенштейна
- Масштабировать в [0, 1]

2. **Найти рынок с максимальной оценкой**:

$$m^* = \arg\max_{m} \text{score}(m)$$

3. **Применить многоуровневую классификацию**:

| Уровень | Условие | Результат |
|---------|---------|----------|
| **Tier 1** (direct match) | $\text{score} \geq 0.65$ | Возвращается $m^*$ |
| **Tier 2** (category-assisted) | $0.40 \leq \text{score} < 0.65$ И Jaccard$(C_{\text{event}}, C_{\text{market}}) \geq 0.30$ | Возвращается $m^*$ |
| **None** | Все остальные случаи | Возвращается None (нет совпадения) |

Где Jaccard (пересечение/объединение):

$$J = \frac{\|C_{\text{event}} \cap C_{\text{market}}\|}{\|C_{\text{event}} \cup C_{\text{market}}\|}$$

### Пороги и параметры

| Параметр | Значение | Описание |
|----------|---------|----------|
| `MARKET_MATCH_TIER1_SCORE` | 65 (0.65) | Пороговая оценка token_sort_ratio для прямого совпадения |
| `MARKET_MATCH_TIER2_SCORE` | 40 (0.40) | Минимальная оценка для Tier 2 |
| `MARKET_MATCH_TIER2_JACCARD` | 0.30 | Минимальное перекрытие категорий для Tier 2 |

### Пример

**Событие:**
- Текст: "ЦБ РФ повысит ставку выше 10%"
- Категории: {"finance", "russia", "central_bank"}

**Рынки в индексе:**
- "CBR interest rate hike above 10%" (score = 0.68, categories = {"finance", "russia"})
- "Russian inflation spike" (score = 0.35, categories = {"finance", "russia"})

**Результат:**
- Рынок 1: score 0.68 ≥ 0.65 → **Tier 1 match** → возвращается
- Рынок 2: score 0.35 < 0.40 → отклоняется

### Фильтрация по индексу

Перед матчингом Judge строит `market_index` из фoresight_signals:

```python
market_index = {
    title.lower(): signal
    for signal in foresight_signals
    if signal.get("source") == "polymarket"
       and signal.get("probability") is not None
       and signal.get("liquidity") >= 10_000
}
```

Только рынки с:
- `source = "polymarket"`
- `probability` определена
- `liquidity >= $10,000`

включаются в поиск.

Исходный код: [`judge.py:513–544`](https://github.com/antopkin/delphi_press/blob/main/src/agents/forecasters/judge.py) и [`fuzzy_match.py:13–66`](https://github.com/antopkin/delphi_press/blob/main/src/utils/fuzzy_match.py).

## Интеграция рыночных сигналов в агрегацию

После того как рынок сопоставлен с событием, Judge интегрирует его в взвешенную медиану:

1. **Расчёт вес рынка**: $w_{\text{market}} = 0.15 \times f_{\text{liq}} \times f_{\text{vol}} \times r_{\text{dist}} \times f_{\text{horizon}}$

2. **Бонус совпадения**: если рыночная вероятность близка к оценке персоны ($|p_{\text{market}} - p_{\text{persona}}| < 0.10$), персона получает бонус:

$$w_{\text{persona}} \mathrel{+}= 0.04$$

3. **Невалидация рынка**: если рыночная вероятность < 0.10, рынок исключается из агрегации

4. **Повторная нормализация**: все веса пересчитываются так, чтобы сумма = 1.0

### Информированные трейдеры (Phase 5)

В Phase 5 добавлена поддержка "informed consensus" — вероятностей от избранных трейдеров с высокой репутацией:

- Если рынок содержит `informed_probability` и `informed_coverage >= 0.30`, Judge использует informed вероятность вместо основной
- Рыночный вес увеличивается на бонус за охват: $w_{\text{market}} \mathrel{+}= 0.05 \times \text{informed_coverage}$

Исходный код: [`judge.py:210–232`](https://github.com/antopkin/delphi_press/blob/main/src/agents/forecasters/judge.py).

## Headline Score и отбор топ-N

После агрегации в timeline Judge ранжирует события по composite score:

$$s_{\text{headline}} = p_{\text{cal}} \times n \times (1 - \text{saturation}) \times r_{\text{outlet}} \times f_{\text{temporal}}$$

Где:

- $p_{\text{cal}}$ — калиброванная вероятность (после Platt scaling)
- $n$ — newsworthiness из консенсуса персон
- $(1 - \text{saturation})$ — штраф за переизбыток в медиа (Stage 3 анализ)
- $r_{\text{outlet}}$ — релевантность события для целевого издания (1.0 по умолчанию)
- $f_{\text{temporal}} = \max(0.5, 1.0 - 0.05 \times |d|)$ — близость к target_date ($d$ — дни)

### Отбор и подстановки

| Параметр | Значение |
|----------|---------|
| `TOP_N_HEADLINES` | 7 |
| `MAX_WILD_CARDS` | 2 |
| `WILD_CARD_NEWSWORTHINESS_MIN` | 0.70 |

Judge выбирает:
1. Топ 7 событий по headline_score
2. До 2 "wildcard" событий (чёрные лебеди контрариана) с newsworthiness ≥ 0.70

Исходный код: [`judge.py:322–646`](https://github.com/antopkin/delphi_press/blob/main/src/agents/forecasters/judge.py).

## Теоретическое обоснование

### Дельфи-метод (Dalkey & Helmer 1963)

Исходный Дельфи-метод был разработан в RAND для прогнозирования будущих технологических событий. Ключевые компоненты:

1. **Анонимность** — участники не видят друг друга, избегая эффекта авторитета
2. **Итеративное уточнение** — несколько раундов позволяют пересмотреть позиции
3. **Медиатор** (в классическом варианте — модератор) — синтезирует расхождения
4. **Аргументированная позиция** — каждый участник записывает обоснование

Наше применение сохраняет все четыре принципа:

- **Анонимность**: посредством deterministic shuffling personas → Expert A, B, C
- **Итеративность**: R1 → медиация → R2
- **Медиатор**: algorithmic + LLM-enhanced synthesis
- **Обоснование**: каждый PredictionItem содержит reasoning, key_assumptions, evidence

### DeLLMphi (Liu et al. 2024)

В недавних работах по LLM-Дельфи (Ling Liu et al., "DeLLMphi", 2024) показано, что синтетические эксперты с явными когнитивными смещениями превосходят простую агрегацию вероятностей. Наша конфигурация персон основана на этом.

Когнитивные смещения в таблице выше ("Переоценивает", "Недооценивает") **намеренны**: они отражают типичные систематические ошибки в аналитике, и их наличие *повышает* качество прогноза, если смещения декоррелированы.

### Substantive Questions (Tetlock 2005)

Метод "substantive questions" требует, чтобы каждый прогноз был привязан к фактическим событиям, а не к мнениям. Это реализуется через:

- `PredictionItem.prediction` — конкретное утверждение, не абстракция
- `PredictionItem.evidence` — ссылки на конкретные факты
- `PredictionItem.key_assumptions` — явные предпосылки для проверки

После разрешения события (вне пайплайна) Brier score вычисляется как:

$$\text{Brier} = \frac{1}{N} \sum_{i=1}^{N} (p_i - o_i)^2$$

где $p_i$ — прогнозируемая вероятность, $o_i$ — исход (0 или 1), $N$ — число событий. Идеальный прогноз имеет Brier = 0.0, случайный угадывающий — Brier = 0.25.

## Часто задаваемые вопросы

### Почему Контрариан имеет T°=0.9 в R1?

Повышенная температура поощряет более творческий поиск альтернативных сценариев и чёрных лебедей. В R2 температура снижается до 0.6 для сведения к обоснованному единству.

### Почему медиатор имеет T°=0.5?

Консервативная температура обеспечивает воспроизводимую, детерминистичную синтез. Медиатор — это *синтезирующий*, а не генерирующий агент.

### Что происходит, если события упомянуты менее чем 3 персонами?

Такие события помечаются как `GapArea` ("пробел"). Они переносятся в R2, но медиатор не вычисляет spread и не классифицирует их как "спор" или "консенсус". Персоны в R2 знают о пробелах и могут переоценить вероятность.

### Как вычисляется spread?

$$\text{spread} = \max(\text{probabilities}) - \min(\text{probabilities})$$

Для 5 персон spread находится в диапазоне [0.0, 1.0]. Пример: если 4 персоны оценивают вероятность в 0.7, а 1 в 0.2, то spread = 0.5.

### Будет ли личность персоны раскрыта в R2?

Нет. Синтез содержит только анонимизированные позиции: "Expert A считает X, потому что Y". Истинный persona_id не видны в `MediatorSynthesis`, переданном персонам.

### Как быть, если нет разногласий?

Если все события имеют spread < 0.15, `MediatorSynthesis.disputes` будет пусто, и R2 будет фокусироваться на `consensus_areas` и `gaps`. Это нормально и указывает на высокую согласованность.

### Может ли персона в R2 полностью изменить свою позицию?

Да. `revisions_made` может содержать перечисление значительных изменений, а `revision_rationale` объясняет причину. Система не навязывает никакого ограничения на величину пересмотра.

### Как работает Platt scaling и почему используется $a = 1.5$?

Platt scaling трансформирует сырую вероятность в калиброванную через логистическую функцию с коэффициентом экстремизации. Параметр $a = 1.5$ означает, что вероятности растягиваются (например, 0.7 → 0.74), делая различия между уверенными прогнозами более выраженными. Это основано на исследовании откалиброванных Дельфи-прогнозов.

### Что происходит, если рынок не найден для события?

Если fuzzy match не найдёт совпадение (обе Tier 1 и Tier 2 не пройдены), Judge просто использует взвешенную медиану персон без рыночного сигнала. Рынок остаётся необязательным.

### Какое значение имеет $f_{\text{temporal}}$ для события за 15 дней до целевой даты?

$$f_{\text{temporal}} = \max(0.5, 1.0 - 0.05 \times 15) = \max(0.5, 0.25) = 0.5$$

Событие за 15 дней получает множитель 0.5 — значительный штраф за удалённость от целевой даты.

### Почему контрариан получает бонус +0.02 для горизонта "near"?

Литература (Tetlock, GJP) показывает, что альтернативные сценарии и риск-анализ наиболее ценны в середине периода неопределённости (7–14 дней). На этом горизонте базовые ставки менее стабильны, и контрариан может обнаружить упущенные риски.
