# Архитектура системы

## Что делает Delphi Press

**Delphi Press** обрабатывает информационные потоки и рыночные сигналы, генерируя прогнозы заголовков для конкретного СМИ на конкретную дату. Система использует мультиагентный конвейер из 18 специализированных агентов, работающих через 9 последовательных стадий, координируемых типизированным контекстом.

## Два режима работы

| Характеристика | Web UI | Claude Code |
|---|---|---|
| **Доступ** | Веб-интерфейс | CLI (`/predict` в Claude Code) |
| **API-ключ** | Пользователь вводит свой (OpenRouter) | Покрыто подпиской Code Max |
| **Стоимость прогноза** | \$0,10–\$15 | \$0 |
| **Разнообразие моделей** | Нет (все Claude Opus 4.6; разнообразие через промпты) | Нет (все Claude Opus 4.6) |
| **Автоматизация** | REST API + расписание | Ручной запуск |
| **Персистентность** | БД, история, аналитика | Markdown-отчёт |

Оба режима разделяют промпты персон и Pydantic-схемы данных.

## Целевая аудитория

Основная аудитория — команды, работающие с прогнозированием событий, медиа-анализом и стратегическим планированием:

- Исследователи-политологи
- Аналитики медиа
- Специалисты по молодёжной политике
- Стратеги государственных учреждений

Интерфейс разработан для нетехнических пользователей с интуитивной системой заполнения форм и визуализацией результатов. Язык интерфейса — русский. Анализ проводится на СМИ на любом языке; результаты генерируются на языке целевого издания.

## Развёртывание: 4 контейнера Docker

Система развёртывается как четыре скоординированных контейнера:

1. **app** (FastAPI/Uvicorn на порту 8000) — веб-интерфейс, REST API, SSE-потоки
2. **worker** (ARQ на Redis) — асинхронное выполнение пайплайна прогнозирования
3. **redis** (порт 6379) — брокер задач, pub/sub для SSE
4. **nginx** (порты 80/443) — reverse proxy, SSL termination via Let's Encrypt

Данные хранятся в SQLite-файле на диске (`/app/data/foresighting.db`). Архитектура выбрана для простоты развёртывания на одном VPS.

**Сервер**: Debian 12, Yandex Cloud, 4 vCPU @ 20%, 8 GB RAM, static IP.

## Принципы проектирования

### Модульный монолит, а не микросервисы

Система построена как единая кодовая база с чёткой модульной структурой. Каждый модуль:

- Живёт в отдельной директории (`src/agents/collectors/`, `src/agents/analysts/` и т. д.)
- Имеет определённый контракт: типизированный вход (Pydantic-модель), выход (`AgentResult`)
- Реализуется и тестируется независимо
- Общается с другими модулями через прямые вызовы функций (внутри одного процесса)

Этот подход избегает сложности микросервисной архитектуры (сетевые задержки, согласованность состояния) при сохранении модульности и тестируемости.

### Async everywhere — асинхронная обработка I/O

Все операции ввода-вывода реализованы асинхронно:

- **httpx** — асинхронный HTTP-клиент для RSS, web search, скрейпинга
- **aiosqlite** — асинхронный драйвер SQLite для БД-операций
- **ARQ** — асинхронная очередь задач на Redis
- **asyncio.gather()** — параллельное выполнение агентов внутри стадии

Вся вычислительная база от app-сервера до worker-процесса построена на asyncio, что позволяет эффективно обрабатывать множество параллельных прогнозов на одном VPS.

### Fail-soft с min_successful логикой

Система не требует, чтобы все агенты в стадии завершились успешно. Вместо этого определяется минимальное количество успешных агентов:

- **Stage 1 (Collection)**: min=2/4 агентов — система может продолжить даже если 2 коллектора отказали
- **Stage 3 (Trajectory)**: min=2/3 аналитиков — достаточно 2 аспектов анализа
- **Stage 4 (Delphi R1)**: min=3/5 персон — мнение 3 экспертов из 5 достаточно для устойчивого консенсуса
- **Stage 5 (Delphi R2)**: min=3/5 (после медиации)

!!! note "Что такое min_successful?"
    Минимальное число агентов, которые должны завершиться успешно для продолжения pipeline. Обеспечивает graceful degradation: если primary-модель недоступна, ModelRouter переключается на fallback-модель через OpenRouter и продолжает работу.

### Разнообразие через промпты и когнитивные смещения

В текущей реализации все 5 персон Дельфи используют **одну и ту же модель** (Claude Opus 4.6 через OpenRouter). Разнообразие ошибок обеспечивается не выбором разных LLM, а тремя механизмами:

1. **Уникальные системные промпты** (~400–500 строк на персону) — каждая персона получает свой аналитический фреймворк, фокус и стиль рассуждений
2. **Когнитивные смещения** — для каждой персоны определены `over_predicts`, `under_predicts` и `anchor_type`
3. **Горизонт-адаптивные веса** — Judge корректирует влияние каждой персоны в зависимости от горизонта прогнозирования

Модельное разнообразие (5 разных LLM) рассматривалось как альтернатива, но не реализовано.

### Pydantic v2 для валидации и структурированного вывода

Все входы и выходы агентов типизированы через Pydantic-модели (`src/schemas/*`):

- **PredictionRequest** — ввод пользователя (выход издания, целевая дата)
- **AgentResult** — выход каждого агента (данные, метрики, ошибки)
- **PipelineContext** — разделяемое состояние (16 слотов)
- **PersonaAssessment**, **RankedPrediction**, **FinalPrediction** — типизированные структуры результатов

**Structured Output**: LLM-ответы не парсятся как сырые строки. Вместо этого OpenRouter запрашивает JSON output, который затем валидируется через Pydantic с помощью `json_mode=true`. Это снижает вероятность парсинга неверного формата.

### Стоимость и бюджетный контроль

Каждый вызов LLM отслеживает:

- Использованные токены (входные/выходные)
- Стоимость в USD (по таблице прайсинга OpenRouter)
- Модель и провайдер
- Время выполнения

Глобальный лимит: **\$50 USD на один прогноз** (настраивается). Если BudgetTracker обнаруживает, что предстоящий вызов превысит лимит, система выбрасывает `LLMBudgetExceededError` и прерывает пайплайн.
